Things to do to pass Deb's Classes

  • Design Notebook

    A note on the design notebook before we move on. I admit in a class of this level it is a bit silly to use (for people who have experience) but for people with little to no experience, it is a very good idea to get started with proper design principles early instead of learning them by neccecaty when a project exceeds their capacity. If you are experienced then play along and roll play as though you were turning a more important design into your boss. Pretend that you actually have a customer and think about it in that perspective and extrapolate what they might want from what they say, you don't have to implement it, just comment on it and think of designs and maybe write a few.

      MUST DOS:

    • Include Tasks. That is read all user stories and requirements and parse that into information on exactly what you need to do and what you don't you are free to speculate and consider additional things, just ensure your provide a well formatted list or flow chart of all functionality of the program broken down into tasks
    • Include Test Cases. provide a nicely formatted chart of expected I/O. Think of it from the stand point of writing an auto grader for your assignment. If you were to write that auto grader, think of everything you would want to check for, what might cause issues, what should throw errors, what should silently handle errors, what should pass what should fail. Best Practice: put breaking cases as the top of the list, IE: any case that might crash or throw an error.
      • Function Prototypes. Include function Prototypes in your designs, that is: what does this function take IE: parameters and what does this function return. Keep in mind passing parameters as constants and making sure you specify type and var name in prototypes.
    • Function Psuodocode. Include psuodocode for every function that you wrote a prototype for. Best practice: Also include some psuodocode for any tasks you didn't write prototypes for, just indicate that you thought about implementing it and changed your mind.
      Note on psuodocode: Deb has a very broad definition of psuodocode unlike some profs who want text only. I recommend including pictures, diagrams or flowcharts mixed into or nearby your psuodocode.
    • Pictures, Pictures, Pictures: Include as many pictures and diagrams as you can to help explain things. MUST: include pictures relating to main's flow and any hard concept, she does not list this explicitly but I strongly strongly, recommend doing so.

  • Coding Style

      MUST DOS:

    • YOU WILL GET A 0% if you use infinite loops
    • DO NOT use break statements
    • DO NOT use multiple returns. (both her and myself are aware there are very good uses for this, stop being a smart ass and follow her style for her class)
      If you need an explanation, she is trying to teach structured programming, and a lot of people use multiple returns as a crutch. Its easier to disallow them for everyone than try to check if people are using them well or not.
    • Documentation Style
      • Documentation Comments.
        Place documentation comments above every function.
        They should provide a brief description, the params, and the return type Documentation comments should be placed above prototypes, I followed DoxGen C standard and she was fine with that. I recommend choosing any popular standardized documentation style and declaring that you are following that standard.
      • Header Comments.
        You should start every c program with a multiline comment at the top of your code with the following:
        • Your Name
        • Class # and section # and the days the class meets IE: CS2060-001(mon/wed)
        • Brief description of the project. Do not use general language, pretend you are writing a real description of a real project. What does this program do? Don't focus on what it will teach you, focus on what it does here.
        • The documentation style you are following IE: Documentation Style: DoxGen C Standard.
        • The Security Standard you are following. Deb uses: Security Standard: SEI CERT C Standard. Don't be fancy and use something else unless you talk to her first
        • (optional) I like to add a note to the grader explaining specific choices, not a neccecaty, but might save you a point or two on occasion.
    • Clearing the Buffer.
      • Use While(getchar()) to clear entire buffer
      • Use While(getchar() != '\n') to clear buffer until newline
      • DO NOT use fflush(stdin) or similar functions. Use getChar or you will lose points
